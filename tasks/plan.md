# Implementation Plan: Deploy CRM to Railway (Free Tier)

## Overview

Deploy the CRM (FastAPI + React + PostgreSQL + Redis + Celery) to Railway's free tier ($5/month credit). The frontend will be built as static files and served directly by FastAPI, eliminating the nginx service. PostgreSQL and Redis will be Railway managed services. Celery runs as a second Railway service. The result is a publicly accessible CRM at a Railway-provided URL (or custom domain).

## Architecture Decisions

- **Single backend Docker image** serves both the API and the built React SPA (via FastAPI's static file serving already coded in `main.py`). No nginx needed in production.
- **Railway** chosen over Render because Render free-tier web services sleep after 15 min inactivity — unusable for a CRM. Railway's $5/month credit keeps services always-on.
- **Railway managed PostgreSQL + Redis** instead of Supabase/Upstash to keep everything in one platform and simplify networking (internal URLs).
- **GitHub** as the deployment source — Railway watches the repo and redeploys on push.
- **CORS origins** loaded from an environment variable so the production URL doesn't require a code change.
- **AWS S3 + SES** remain as-is (already configured, low-cost AWS services).

## Dependency Graph

```
Railway PostgreSQL (provisioned first)
    │
    └── Alembic migrations (run once, manually)
            │
            └── FastAPI backend (needs DB + Redis)
                    │
                    ├── React frontend (built into backend/static/ at Docker build time)
                    │
                    └── Celery worker (needs DB + Redis)

Railway Redis (provisioned alongside PostgreSQL)
    │
    ├── FastAPI backend
    └── Celery worker
```

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Railway $5 credit exhausted | High | ~2 services + DB + Redis ≈ $3-4/mo on free tier; monitor usage |
| Redis 25MB limit on Railway free | Medium | Celery results are small; purge old task results periodically |
| Celery task loses S3 file path if container restarts | Low | S3 key stored in DB before task starts |
| Cold start latency on Railway | Low | Railway free tier keeps services warm (unlike Render) |
| alembic migration fails on Railway DB | Medium | Test migration locally against a temp Postgres first |
